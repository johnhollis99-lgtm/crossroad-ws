# v1.1 polygon followups — regions accepted with inadequate v1 polygon

**Created:** 2026-05-14
**Updated:** 2026-05-14 (Phase 4 post-import audit — added Cuyama/Panamint/Yosemite osm_linear_to_bbox cases)
**Phase:** E1d (named valleys/basins) — Phase 2 polygon-source verification + Phase 4 live import
**Status:** Open. Address before v1.1 release.

## Context

Phase E1d Phase-2 polygon verification ran four iterations (OSM name-match → location-sanity check → tag-fallback by AVA/desert/region/admin → Wikidata centroid+buffer). For two regions in the top-30, no usable OSM polygon was found via any path. Per curator direction (2026-05-14), these were accepted at Tier B with the Wikidata 15km heuristic buffer and tagged `metadata.polygon_quality = 'inadequate_buffer_v1'` so the limitation is auditable.

Phase-4 live import (commit 9418afd) surfaced a second class of polygon-quality issue: three OSM `natural=valley`-tagged ways turned out to be unclosed LineStrings (representing the valley's central axis, not a closed boundary polygon). Per the script's bbox-rectangle fallback, these were inserted with a rectangular polygon spanning the way's bounding box. Tagged `metadata.polygon_quality = 'osm_linear_to_bbox_v1'`.

A 15km circular buffer = ~707 km². For ~250 km² landforms (e.g., Anderson Valley, Capay Valley) the buffer is generous; for the buffer-inadequate regions below, the buffer is 1–10% of the real region area, so narration will only trigger near the geographic centroid rather than across the actual region.

A bbox-rectangle approximation over a linear OSM feature is geographically generous (overcompasses the valley with rectangular sides) but loses the actual valley outline. Acceptable for v1; needs proper polygon in v1.1.

## Why each one failed automated resolution

### San Joaquin Valley (rank 16)
- OSM name-match: zero relations or ways named "San Joaquin Valley" with polygon-relevant tags in CA.
- Tag-fallback (`place=region`, `boundary=region`, `boundary=administrative[admin_level=6]` within 50km of Wikidata centroid): no candidate. Centroid is in the middle of the valley; no single county polygon is dominant there, and no `place=region` named for the valley.
- Final v1: Wikidata Q964198 centroid + 15km heuristic buffer.

### Los Angeles Basin (rank 27)
- OSM name-match: zero usable matches.
- Tag-fallback: found `relation/17463174 Southern California (place=region)` with bbox area 244,313 km². Rejected by area-plausibility check — ratio 20.36 vs real area 12,000 km² is outside [0.1, 10] bounds. Polygon was clearly representing all of SoCal, not the LA Basin specifically.
- Final v1: Wikidata Q2887490 centroid + 15km heuristic buffer.

## Recommended v1.1 paths to investigate

For both regions:

1. **Editorial polygon (Tier C path).** Most reliable. Hand-digitize from terrain map at ~10km tolerance — these are large landforms, precision isn't load-bearing. Estimated ~30 min each for an experienced digitizer.

2. **Composite-of-counties.** Build polygon as `ST_Union(county_polygons)` for the counties that comprise the valley. SJV = Kern + Tulare + Kings + Fresno + Madera + Merced + Stanislaus + San Joaquin counties. LA Basin = LA County + parts of Orange/Ventura. Pros: cheap, can be automated from OSM admin-level=6 county geometries. Cons: includes mountainous terrain that isn't part of the valley landform (e.g., Sierra Nevada eastern flank for SJV).

3. **Watershed approximation.** For SJV: union of HUC8 watersheds in the valley floor (USGS data, public domain). Cleaner geological boundary than admin-by-county. Requires HUC8 data import (Phase E1e watersheds work anyway).

4. **EPA Level III Ecoregion overlay.** Phase E1b already imported EPA ecoregions. SJV roughly corresponds to L3 ecoregion 7 (Central California Valley); LA Basin partially overlaps L3 ecoregions in the Southern California Mountains family. Could derive a "valley region" by intersecting EPA ecoregion polygons with elevation bands.

My recommendation: option (1) for both. It's the most explicit and avoids cross-layer dependencies. Phase E1e watershed work can validate the SJV polygon when it lands.

## Schema/tagging requirement

When v1 polygons are inserted to `regions`, tag `metadata.polygon_quality = 'inadequate_buffer_v1'` so a future maintenance query can find them:

```sql
SELECT id, name FROM regions
WHERE metadata->>'polygon_quality' = 'inadequate_buffer_v1';
```

That query is the v1.1 work list.
